# AGENTS.md

This repository is a Puppet module bootstrapped with PDK.

## PDK-Managed Files

Many files in this repository are managed by PDK through [.sync.yml](.sync.yml)
and `pdk update`.

- Do not modify PDK-managed files directly.
- Prefer changing `.sync.yml` or the appropriate source inputs, then run
  `pdk update` when PDK-managed content needs to change.
- If a file appears templated or generated by PDK, treat it as managed unless
  you have confirmed otherwise.

## Validation And Testing

Primary verification should use PDK tooling:

- `pdk validate`
- `pdk test unit`

Run the relevant PDK checks after changes whenever feasible, and prefer these
over ad hoc validation.

The CI configuration in this repository runs `pdk validate` and
`pdk test unit --parallel --verbose`. Avoid running multiple separate `pdk`
commands at the same time, but the unit test command's own `--parallel` mode is
part of the normal workflow here.

## Commit Conventions

- Git commit subjects should follow the pattern: `component: Short description`
- Commit bodies should be wrapped and explain the relevant details and intent.
- When creating commits with multi-line bodies, use `git commit -F <file>` (or
  `git commit --amend -F <file>`) instead of `-m` with escaped newlines.
- All commits must be signed with the user's valid SSH signature.

## Working Notes For Future Agents

- Start by checking whether intended edits touch PDK-managed files.
- Avoid bypassing the PDK workflow for templated files.
- Prefer unit tests and validation over running live deployment flows.
- [`plans/deploy.pp`](plans/deploy.pp) shells out to local `helm` and `kubectl`
  commands, so treat plan execution as potentially environment- and
  cluster-affecting.
- When reporting verification, call out which PDK commands were run and whether
  anything was intentionally skipped.
